Case Study 1: Supply Chain & Procurement

Scenario: Kanadevia is constructing a massive waste-to-energy plant. Critical materials (steel, turbines, electronics) are delayed. The Project Manager asks: "Which suppliers are high-risk, and how does this impact our budget?"

Goal: Analyze On-Time Delivery (OTD) and Spend Volume by Category.

Supplier Analysis: OTD % vs. Spend

Insight: The "Turbines" category has high spend but low OTD (65%). This is a critical risk for the project timeline.

DAX Solution (Key Metric)

During the interview, they might ask how to calculate dynamic delivery percentages.

OTD % (On-Time Delivery) = 
VAR TotalOrders = COUNTROWS('PurchaseOrders')
VAR OnTimeOrders = 
    CALCULATE(
        COUNTROWS('PurchaseOrders'),
        'PurchaseOrders'[DeliveryStatus] = "On Time"
    )
RETURN 
    DIVIDE(OnTimeOrders, TotalOrders, 0)

// Extension: Year-over-Year (YoY) Change
OTD YoY Change = 
VAR CurrentOTD = [OTD %]
VAR PreviousOTD = 
    CALCULATE(
        [OTD %], 
        SAMEPERIODLASTYEAR('Calendar'[Date])
    )
RETURN 
    CurrentOTD - PreviousOTD

Case Study 2: Project Cost Control (EPC)

Scenario: The CFO needs to see the "Burn Rate" of the budget for a construction project in the UK. We need to compare Planned (Budget) vs. Actuals and forecast the final outcome.

Cumulative Budget Variance

Insight: In Q3, we exceeded the budget due to material price increases. A forecast revision is required.

Advanced DAX Techniques

Using CALCULATE with filter modifiers and Time Intelligence patterns.

Cumulative Cost = 
CALCULATE(
    SUM('ProjectCosts'[Amount]),
    FILTER(
        ALL('Calendar'),
        'Calendar'[Date] <= MAX('Calendar'[Date])
    )
)

Cost Variance % = 
VAR Actual = [Total Actual Cost]
VAR Budget = [Total Budget Cost]
RETURN 
    DIVIDE(Actual - Budget, Budget, 0)

Big Data Approach

  • Aggregation: We don't import every second into Power BI. Use SQL views to aggregate averages per 15 minutes (Granularity reduction).
  • DirectQuery: For real-time monitoring, use DirectQuery to Azure SQL or Azure Data Explorer.
  • DAX Moving Average: To smooth out data noise.
7-Day Moving Avg Temp = 
AVERAGEX(

Common Q&A Questions

Q: Difference between Calculated Column and Measure?

Measure: Calculated "on the fly" during report interaction, dependent on filter context (CPU intensive). Calculated Column: Calculated during data refresh, stored in RAM model (Storage intensive). For a company like Kanadevia (Big Data), we prefer Measures.

Q: What is a Star Schema and why use it?

It is a modeling standard with one central Fact Table (transactions, measurements) surrounded by Dimension Tables (Customers, Time, Material). It ensures faster reporting and simpler DAX formulas compared to a single flat table.

Q: How do you optimize a slow PBI report?

1. Check Performance Analyzer.
2. Remove unnecessary columns and rows (VertiPaq engine).
3. Replace complex DAX filters (FILTER vs KEEPFILTERS).
4. Move transformations to Power Query or SQL (Query Folding).

Q: What is Row Level Security (RLS)?

A security feature that restricts data access based on the user. E.g., The UK Manager sees only "Region = UK" data, while the Global Head sees everything. Defined in PBI Desktop via DAX roles.
